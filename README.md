# Исправленная логика

1. **Лимиты работают вместе:** максимум 5 разных товаров, каждого от 1 до 10 штук, общее количество не больше 20
2. **Удаление:** уменьшение количества до 0 удаляет товар, отдельная кнопка тоже есть
3. **Цена:** фиксируется при добавлении (чтобы сумма не менялась у пользователя)
4. **Ошибки:** показываем конкретную причину ("не больше 10 штук одного товара" и т.д.)

# Задание 3: Архитектура PUSH-уведомлений
Схема работы
flowchart TD
    A[Мобильное приложение] -->|Регистрация токена| B[API Gateway]
    
    C[Сервис заказов] -->|Событие| D[Kafka]
    E[Сервис пользователей] -->|Событие| D
    F[Рекламный сервис] -->|Событие| D
    
    D --> G[Notification Service]
    G --> H[(БД токенов)]
    G --> I[Очередь задач]
    
    J[Worker] --> I
    J --> K{Платформа}
    K -->|iOS| L[APNS]
    K -->|Android| M[FCM]
    
    L --> A
    M --> A

# Как это работает
Мобилка при запуске отправляет токен устройства (регистрируется)

Микросервисы (заказы, пользователи, реклама) при нужном событии кидают сообщение в Kafka

Notification Service ловит событие, берёт из БД все токены пользователя

Кладёт задачи в очередь (Redis/Kafka)

Worker забирает задачи и отправляет в APNS (iOS) или FCM (Android)

Статус доставки сохраняется в БД для аналитики

Пример сообщения в очереди

{
  "user_id": 12345,
  "device_token": "token_ios_123",
  "platform": "ios",
  "payload": {
    "title": "Забыли про заказ?",
    "body": "В корзине товаров на 3 450 ₽",
    "deep_link": "petrushka://cart"
  }
}

# Важные моменты
Токены протухают — если APNS/FCM вернул ошибку, удаляем из БД

Ретри — при ошибках сети пробуем отправить ещё 3 раза

Не спамить — настраиваем лимиты на количество пушей в день
